eMule broadband edition, compactly eMule BB, is an independent product line for people who still value eMule's distributed sharing model. It preserves the classic desktop workflows while making the client easier to run, observe, automate, and validate on current Windows systems.

Feature list

Broadband features for power users and eMule aficionados

The headline features are the ones operators feel first: fewer bad upload slots, better large-library behavior, and an API surface that lets companion tools cooperate with the desktop client instead of pretending it is a generic downloader.

Upload control

Bounded broadband slots

Uses a configured upload budget and target slot count instead of letting slot count drift toward legacy high-slot behavior on fast links.

Queue quality

Weak-slot recycling

Detects slow or stuck uploaders after warm-up, requeues them with cooldown, and gives waiting clients a cleaner chance to fill the pipe.

Seeder tools

Ratio-aware sharing

Adds all-time and session ratio readouts, low-ratio file boost controls, and optional LowID score deboosting for stricter seeders.

Large libraries

Safer shared-file handling

Focuses on recursive share sync, startup cache behavior, long-path Windows libraries, stable sorting, and responsive large collections.

Classic network

eD2K and Kad first

Keeps server/global/Kad search modes, shared files, upload queues, categories, known clients, and desktop control as the product foundation.

Operations

Safer long runs

Improves persistence, shutdown behavior, WebServer validation, background worker lifetime, and error paths that matter in always-on sessions.

Automation

REST control surface

Exposes authenticated JSON endpoints for transfers, searches, shared files, servers, Kad, status, logs, categories, and controlled shutdown.

Controller fit

aMuTorrent and Arr workflows

Targets practical local integration: search, add, observe, and manage eMule from companion tools while the native app remains in charge.

Release process

Built to be proven

Uses native tests, Python harnesses, UI automation, REST smoke checks, live eD2K/Kad scenarios, and controller integration gates.

Product guide

A short operating model

01

Start from known-good eMule habits

Use trusted server lists, bootstrap Kad deliberately, keep incoming and shared directories predictable, and preserve the classic search/add/share workflow before layering automation on top.

02

Tune upload for your real link

Set a finite upload limit, choose a realistic upload client target, and let the broadband policy favor fewer stronger slots instead of a hundred low-rate sessions.

03

Curate large libraries

Use long-path capable Windows setups, keep share roots clean, watch all-time and session ratios, and treat rare files as deliberate publishing decisions.

04

Automate only on trusted networks

Enable the WebServer/REST surface with an API key, bind and firewall it carefully, and use controllers that respect native eMule transfer and delete semantics.

05

Track release readiness

Treat the public branch as active pre-release work until Release 1 gates, operator checks, and live E2E evidence say otherwise.

06

Use the docs for depth

The homepage stays compact. The detailed product guide, REST contract, broadband notes, and release documents live as Markdown in the public tooling repository.

Controller surface

REST automation without replacing the desktop app

The broadband release track exposes a resource-oriented /api/v1 JSON API from the existing WebServer listener. It authenticates with X-API-Key, serves JSON envelopes, and keeps native eMule state changes marshaled through the app.
